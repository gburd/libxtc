# M4.5 — fast fiber context-switch ASM (deferred)

**Status:** Deferred from the M4.5/M5.5/M7 round.  Fiber switches
currently use POSIX `ucontext` (~600 ns per switch on x86-64 glibc).
This document records the plan and why we chose to wait.

## Why deferred

Implementing `make_fcontext` / `jump_fcontext` from scratch is a
common foot-gun — the stack layout and register save/restore have to
match exactly between the two functions, with very subtle alignment
constraints.  Getting the asm wrong silently corrupts the saved
register file and produces test failures hours later, in non-trivial
patterns.

The known-correct path is to vendor the **Boost.Context** asm files
(BSL-1.0, GPL/PG-compatible) which are battle-tested across every
arch we care about.  The vendor process needs:

1. Pick the upstream Boost.Context revision to import.
2. Copy `src/asm/{make,jump}_*_sysv_elf_gas.S` for our 12 archs:
   `x86_64`, `i386`, `arm64-aapcs-elf`, `arm-eabi`, `ppc64le`,
   `s390x`, `riscv64`, `sparc64`, `mips64`, `loongarch64`, plus
   the macOS variants (`*_sysv_macho_gas.S`) and the Windows MS
   ABI variants for x86_64 + arm64.
3. Add a thin C header (`fctx.h`) declaring the two functions and
   the `transfer_t` struct.
4. Wire `evt_fiber.c` to call them when `XTC_HAVE_FIBER_ASM` is set
   at configure time.
5. Add a per-arch micro-benchmark that round-trips a no-op context
   switch 10⁷ times and expects ≤ 50 ns/switch.

## Why not write it ourselves

We tried writing `x86_64_sysv` from scratch in one pass during M5.5
and produced a draft that *looked* right but failed the round-trip
test.  Rather than ship subtly-broken asm we keep the ucontext path
and tag this as deferred.

The cost of waiting: 600 ns per context switch instead of ~30 ns.
That matters for hot-path coroutines (per-connection servers
churning fibers per request) but does **not** affect correctness.
Every M4 test passes today on ucontext.

## What to do now (the placeholder)

`src/os/asm/` is empty.  When M4.5 lands:

- Add the per-arch `.S` files.
- Add `dist/aclocal/ax_fctx_arch.m4` to detect the arch and pick
  the right asm.
- Add `XTC_HAVE_FIBER_ASM` to `xtc_config.h.in`.
- Update `evt_fiber.c` to use the asm when present, ucontext as a
  fallback.

## Triggers for actually doing it

This work jumps the queue when:

- A profiled benchmark shows context-switch overhead is the
  bottleneck for a target workload (PG-on-xtc backends doing
  coroutine-heavy I/O).
- We have at least one BSD VM in the test harness so we can verify
  the macOS / *BSD asm at the same time.
- Someone from upstream offers a clean import of Boost.Context's
  asm with the right license header.
